# Glossary

Universal terminology for `lightassay`. These terms must be used consistently across all documentation, code, and prompts. Do not use domain-specific or application-specific substitutes.

---

## Core terms

### Workflow under test
The applied LLM workflow being evaluated. This is the system whose behavior the user wants to understand. It may be a single LLM call, a multi-step pipeline, an agent, or any other LLM-based process.

### Evaluation session
One complete cycle from brief through analysis for a specific workflow under test. Represented by a single workbook file.

### Target
The first-class definition of what is being evaluated. It identifies the target kind, name, locator, real execution boundary, source references, and optional scope notes. The target is the canonical starting point for planning.

### Target sources
The files or modules that preparation must inspect to build source-grounded directions and cases. They anchor bounded discovery; they are not permission to scan the whole repository.

### Workbook
A single markdown file that is the source of truth for one evaluation session. It contains the brief, directions, cases, human feedback, readiness status, and references to produced artifacts. It is overwritten as the current working state — not versioned as a history.

### Brief
The guided free-form expression of the user's testing intention, written inside the workbook. It captures what to test, what matters, what failure modes are important, and any constraints. It is written in natural language and is internally structured into user priorities. It is a primary planning input, but not the only one: planning is also grounded in the target and source context.

### User priorities
The structured internal representation of the human brief. User priorities preserve what the human explicitly said matters most. They must remain primary even when planning is enriched by code- and prompt-grounded reasoning.

### Planning-ready
The state where a workbook has enough information to plan directions and cases. In v1 this requires a filled target, at least one target source, and human-authored brief content.

### Run-ready
The stricter state where a workbook is ready for execution. It requires the workbook to be planning-ready, directions and cases to exist, workbook `RUN_READY: yes`, and a valid execution binding such as a workflow config with a usable adapter or driver.

### Source-grounded planning
Preparation that uses target, target sources, and bounded target-anchored discovery to understand the code and prompts behind the workflow before generating directions or cases. This is white-box-informed planning; execution still remains black-box at the target boundary.

### Direction
A high-level testing axis produced by the LLM from the target, source-grounded context, and user priorities. Directions are expressed in two universal coordinates: a **behavior obligation** and a **test lens**. They are not test cases — they are a map of the testing space.

### Behavior obligation
A high-level responsibility of the workflow under test (e.g., "produce the main decision", "explain the output", "preserve required constraints"). Used as one coordinate when generating directions.

### Test lens
A universal way of looking at quality (e.g., "nominal correct behavior", "known failure modes", "boundary cases", "robustness under noisy input", "must-not-break invariants"). Used as the other coordinate when generating directions. Not a case — a perspective.

### Case
A concrete test instance generated by the LLM from directions, user priorities, and source-grounded target understanding. Each case has a stable ID, an input, optional context, notes, target directions, and an expected behavior description.

### Expected behavior
A structured semantic description of what the workflow under test should do for a given case. It describes intent, not a code contract. It is used by the LLM during analysis to reason about whether the workflow performed as intended.

### Human feedback
Instructions left by the human in designated fields within the workbook. Human feedback is never written into source content directly — it lives in separate `HUMAN:global_instruction` and `HUMAN:instruction` fields. An empty field means "no objections, approved."

### Run readiness
The LLM-assessed and human-visible status of whether the workbook is ready for a run. Expressed as `RUN_READY: yes` or `RUN_READY: no` in the workbook. This is an orchestration signal, not a quality judgment by code.

### Quickstart
The main self-serve entrypoint. Takes one plain-language message plus a target hint and drives the full bootstrap → preparation → run → analysis pass. Writes a canonical workbook and an analysis artifact. `compare` is deliberately not part of quickstart.

### Continue
The iterative counterpart of quickstart. Runs one full next iteration on the active workbook: extends or refines directions and cases, runs again, analyzes again, and optionally compares against the prior run. Consumes the workbook's current continuation fields and/or a `--message`; either or both are allowed.

### Continuation block
The `## Continue Next Run` section at the top of the workbook. Holds the current editable continuation fields (general instruction, direction instruction, case instruction) plus versioned historical entries of every successful continue iteration.

### CLI message
The literal `--message` argument supplied to `continue`. Stored in each historical continuation entry as `### Continuation v<n>: CLI message` so the workbook records exactly how each iteration was requested. Empty when no `--message` was supplied.

### Full-intent mode
Quickstart mode enabled by `--full-intent`. Disables the default minimal narrowing on suite breadth when the human request genuinely asks for a non-minimal first pass. Human priority already wins in baseline mode — this flag only turns off the internal minimal-first-pass shaping.

### Bootstrap adapter
Adapter operation (`operation: bootstrap`) invoked by quickstart. Receives the user message, the target hint, and the workspace root; returns a structured target, execution shape, assumptions, and planning constraints — or a single `clarification_request` when the target cannot be safely resolved. Local code never auto-binds hints to execution shapes.

### Agent
The user-facing term for the built-in planning/analysis integration used by
the CLI. Currently `claude-cli` and `codex-cli`. Selected via `init`,
`agents`, or `--agent <name>` on `quickstart` / `continue`.

### Backend
The internal architectural term for the named bundle of preparation and
semantic adapters shipped under `lightassay.builtin_adapters`. The
user-facing CLI term for a backend is **agent**. Backends do not carry
workflow-under-test LLM metadata.

### Active workbook pointer
A single-source state file at `{cwd}/.lightassay/active_workbook.json` that records the last workbook touched by `quickstart` or `continue`. Scoped to the current working directory, never mirrored. A corrupt pointer file raises an explicit error instead of silently behaving as "no pointer".

### Workbook id
The canonical identifier of a known workbook, derived from its filename stem (`my-eval.workbook.md` → `my-eval`). Stored alongside the active pointer in `{cwd}/.lightassay/workbooks.json` and used by `continue --workbook-id` and the `workbooks` command.

### Execution log
A JSONL file at `{cwd}/.lightassay/execution_log.jsonl` that records one line per stage transition (`bootstrap`, `preparing directions`, `run`, `analyze`, `compare`, `completed`). Written for debugging and traceability — not intended for default stdout.

### LLM metadata
Optional `{provider, model}` annotation on a workflow config that names the LLM under test. Separate from the execution binding — callable / HTTP / command workflows need no placeholder values.

### Quick try
A bounded planning mode that creates a real workbook, records explicit assumptions, and generates exactly one representative direction and one representative case. It is a bridge into the full workbook model, not a separate simplified format.

### Refinement workbook
A new planning workbook created from an existing suite. It preserves the target and keeps prior directions and cases as first-class structure while adding explicit refinement context.

### Exploratory workbook
A new bounded planning workbook created from an existing suite and a seed run artifact. It is used for iterative investigation of weak spots under explicit case and iteration limits, with a fresh run artifact produced on each exploratory iteration.

### Run
One independent execution of the workflow under test against all approved cases. Each run is self-contained. There is no partial run or resume in v1. If a run fails, the cause is fixed and a new run is started from scratch.

### Run artifact
A machine-readable JSON file produced by a completed or failed run. It contains all raw execution facts: inputs, raw responses, parsed responses (if any), timestamps, token usage, latency, and per-case and run-level status. It is the source of truth for analysis and compare.

### Run status
The top-level outcome of a run. In v1: `completed` (all cases executed without execution failure) or `failed` (at least one case had an execution failure).

### Case execution status
The outcome of executing a single case within a run. In v1: `completed` or `failed_execution`. No quality judgment is embedded in the status.

### Analysis
A separate semantic operation performed by an LLM on a single run artifact. It produces an analysis artifact (markdown). Analysis identifies where the workflow met expectations, where it did not, patterns, weak spots, and recommendations. Code does not perform analysis.

### Analysis artifact
A markdown document produced by analysis. It represents the LLM's semantic interpretation of one run artifact.

### Compare
A separate semantic operation performed by an LLM on two or more completed run artifacts. It is never part of a run. It produces a compare artifact (markdown). Compare is always initiated explicitly after runs are complete.

### Compare artifact
A markdown document produced by compare. It shows which run configurations were compared, where one variant outperformed another, different failure patterns, raw fact differences, and recommendations where appropriate.

### Workbook SHA-256
A SHA-256 digest of the workbook file at the moment a run starts. Stored in the run artifact as `workbook_sha256`. This is the only link between a run artifact and the specific workbook state it was based on. No other history mechanism is required.

## Role terms

### Human
The person who defines the testing intention, sets constraints and priorities, provides human feedback, and interprets final results.

### LLM
The language model used for semantic tasks: generating directions and cases, reconciling the workbook, performing analysis, and performing compare.

### Code
The tool's own logic: guiding the flow, calling the workflow under test, writing artifacts, and measuring raw execution facts. Code never judges quality.

---

## What these terms are NOT

- **Direction** is not a test case and not a test script.
- **Target sources** are not permission for unbounded repository exploration.
- **Expected behavior** is not a coded assertion or a regex pattern.
- **Run status** is not a quality score.
- **Analysis** is not an automated metric.
- **Compare** is not a diff of strings.
- **Workbook** is not a database record.
